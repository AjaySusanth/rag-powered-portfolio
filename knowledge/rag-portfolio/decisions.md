# RAG-Powered Developer Portfolio - Technical Decisions Log

## Why Hybrid Search was Selected over Vector-Only Search in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, a hybrid search combining pgvector semantic search and BM25 lexical keyword matching was chosen. Pure vector search is highly effective at capturing general concepts and synonyms, but it is notorious for failing to locate exact matches for system components, variable names, or technical terms like `JWT`, `AKS`, `Calico`, or `ArgoCD`. During early testing, a query asking about "how is JWT authentication configured in ClassSync" failed to retrieve the auth controller because the semantic distance between the search terms and the raw code block was too wide. Lexical search models like BM25 excel at matching precise keywords but miss semantic synonyms. By running both in parallel and combining them via Reciprocal Rank Fusion (RRF), the portfolio achieves both semantic coverage and exact code/config matching. The tradeoff is the CPU and memory cost of maintaining a BM25 index in memory alongside PostgreSQL, but this is offset by the dramatic improvement in retrieval accuracy for code-heavy queries, which is a core requirement of this engineering portfolio.

## Why Redis Caching was Implemented as a Dual Caching Architecture in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, a dual-layer caching strategy powered by Redis 7 was chosen instead of a single query-level response cache. The first layer is an Embedding Cache, which stores MD5 hashes of text strings mapped to their 1536-dimensional Azure OpenAI embedding vectors with a 7-day Time to Live (TTL). The second layer is a Response Cache, which maps query text hashes and their corresponding retrieved chunk IDs to the generated markdown response with a 24-hour TTL. A single query cache is brittle; a user making a minor typo or formatting change would bypass the cache and trigger a costly embedding generation and LLM pass. By caching embedding vectors separately, identical sentences or common queries avoid redundant Azure OpenAI API calls even if they miss the response cache. Furthermore, the response cache is automatically invalidated and flushed whenever a new `/ingest` run finishes. This design minimizes API costs and keeps latency under 3 seconds.

## Why Gemini Flash was Chosen as the Primary Generation and Grading Model in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, Google Gemini Flash was selected as the primary LLM for answer generation, retrieval grading, query rewriting, and citation attribution. While models like OpenAI's GPT-4o offer excellent reasoning capabilities, their API costs are high for a personal portfolio. Alternatively, open-source models like Llama-3.1-70b hosted on Groq offer fast inference times but are subject to strict rate limits. Gemini Flash, specifically configured for `gemini-3.1-flash-lite` or `gemini-2.0-flash`, offers a massive context window and extremely fast token generation speeds. Furthermore, the Google AI Studio free tier provides a generous quota of 1500 Requests Per Minute (RPM) and 1.5 Million Tokens Per Minute (TPM), which makes hosting the portfolio project cost-effective. To handle potential API downtime, the system implements a provider-agnostic factory that falls back to Groq llama-3.1-70b if Gemini API requests fail.

## Why an Offline Evaluation Framework and Golden Dataset were Developed in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, an offline evaluation framework using a structured golden dataset of question-answer pairs was developed. Tuning a RAG pipeline—such as adjusting chunk sizes, token overlap, RRF parameters, or grading thresholds—without metric baselines makes it impossible to know if changes improve or degrade retrieval. To resolve this, we created a evaluation framework that computes metrics like Hit Rate@K, Recall@K, and Mean Reciprocal Rank (MRR) by comparing retrieved chunks against manually labeled ground-truth files. For example, evaluations demonstrated that introducing the Source Diversification stage increased the Average Unique Sources@5 from 4.40 to a perfect 5.00 and improved MRR from 0.456 to 0.467. Running these evaluations locally prevents regressions before deployment.
