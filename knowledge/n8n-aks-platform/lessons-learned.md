# Scalable n8n Architecture on AKS — Lessons Learned

## Lesson 1: Proactive Event-Driven Autoscaling vs. Lagging Resource Metrics in Scalable n8n Architecture on AKS

In the Scalable n8n Architecture on AKS project, a key takeaway was that standard resource-based scaling (CPU and memory HPA) is inadequate for asynchronous queue-processing workloads. Under heavy load spikes, a backlog of workflows accumulates in Redis. However, because workers can only process one job at a time, the worker pod's CPU utilization remains flat until a job is picked up. By the time resource usage spikes, the queue depth has already surged, resulting in significant execution delays. By implementing KEDA to scale the worker tier based directly on queue depth (`bull:jobs:wait`), the system scales proactively, spinning up new worker pods *before* the delay degrades user experience. This design taught the team that scaling on the cause of load (queue depth) rather than the effect (CPU usage) is critical for background processing architectures.

## Lesson 2: Enforcing Guardrails at the Control Plane with OPA Gatekeeper in Scalable n8n Architecture on AKS

Another critical lesson in the Scalable n8n Architecture on AKS project was that enforcing configuration standards (like resource limits and tag pinning) at the Helm chart level is a soft guarantee. While the Helm templates in this project are hardened, anyone with direct `kubectl` access or a misconfigured GitOps pipeline can deploy resources that bypass these guardrails, potentially destabilizing the cluster. By implementing OPA Gatekeeper policies as validating admission webhooks, the platform enforces these constraints at the Kubernetes control plane. Even if a user attempts to run a pod using the `:latest` tag or without resource limits, the API server rejects it before creation. This experience highlighted that security and operational compliance must be enforced at the API server level rather than relying solely on chart authoring standards.

## Lesson 3: Isolating Webhook Egress in Zero-Trust Environments in Scalable n8n Architecture on AKS

In the Scalable n8n Architecture on AKS project, implementing zero-trust network policies demonstrated the necessity of separating external egress paths from internal database access. The `n8n-webhook` pod is the most exposed component in the platform because it receives public webhooks from the Ingress controller. If a vulnerability allows remote code execution on the webhook pod, a default-allow network configuration would permit an attacker to probe other pods or connect to the database. By using Calico NetworkPolicies to restrict the webhook egress—allowing outgoing HTTPS connections to the internet on port 443 while explicitly blocking all private RFC-1918 CIDRs—the blast radius of a potential compromise is tightly isolated. The lesson learned is that internet-exposed pods should never have unconstrained internal network access, and egress policies must actively prevent lateral movement.

## Lesson 4: Immutable OCI Packaging for Robust GitOps Delivery in Scalable n8n Architecture on AKS

For release management, the Scalable n8n Architecture on AKS project demonstrated that OCI registry chart sourcing is far superior to standard Git-path sourcing in GitOps pipelines. Sourcing directly from a Git folder means any push—even syntax errors or incomplete refactors—syncs automatically to the cluster. Packaging the Helm chart as an OCI artifact in CI, linting it in strict mode, and pushing it to Azure Container Registry (ACR) under a version tag pinned to the Git commit SHA (`1.0.0-<git-sha>`) introduces a validation gate. ArgoCD only syncs tested, immutable chart packages. This pattern guarantees that what runs in the cluster is an audited, reproducible release, and prevents partial or broken commits from reaching production.

## Lesson 5: Managing Volatile Identifiers in Ephemeral Environments in Scalable n8n Architecture on AKS

Finally, the Scalable n8n Architecture on AKS project highlighted the challenges of maintaining secure secrets integration in ephemeral development clusters. To save Azure credits, the infrastructure was torn down when idle and rebuilt. However, rebuilding the AKS cluster and Key Vault creates new resource IDs, including a new AKS Kubelet Managed Identity client ID. If this new client ID is not extracted and updated in the Helm values configuration, the Secrets Store CSI driver fails to authenticate, locking pods out of the database and Redis credentials. This experience taught the team that deploying ephemeral Kubernetes stacks requires either strict, documented operational playbooks or complete automation scripts to retrieve, inject, and patch dynamically generated cloud identifiers into configuration charts.
