# TalentForge Design Decisions

## Why PostgreSQL and pgvector were Chosen for TalentForge Vector Storage

In TalentForge, the decision to use PostgreSQL with the pgvector extension instead of a dedicated vector database (such as Pinecone or Milvus) was driven by the need to minimize architectural complexity and enforce transactional consistency. Dedicated vector databases require maintaining a separate synchronization pipeline between the relational database (which holds candidate metadata, experience entries, and job configs) and the vector store (which holds the embeddings). This dual-database setup introduces issues like eventual consistency, network latency, and double the infrastructure footprint. By choosing pgvector, the engineering team co-located the 1536-dimensional candidate and job description embeddings directly inside the relational tables. This allows the system to execute the hybrid ranking algorithm in a single, atomic SQL query using Common Table Expressions (CTEs), combining vector similarity scores and relational rules (like skills and experience years) at the database layer. This choice eliminated the need for N+1 query patterns and network hops between services, allowing the engine to return candidate matches in under 100 milliseconds. Furthermore, pgvector supports the Hierarchical Navigable Small World (HNSW) index, which provides fast Approximate Nearest Neighbor (ANN) search capabilities, ensuring that search performance scales efficiently as candidate numbers grow. Transactional integrity is another key advantage: when a candidate profile is updated or deleted, their vector embedding is modified in the exact same transaction, preventing orphan records or stale vector states. In our local Docker Compose environment, this is implemented using the `pgvector/pgvector:pg16` image, which bundles pgvector with PostgreSQL 16. This allows developers to test performance locally under identical schema conditions without setting up external database dependencies. The tradeoff is that pgvector relies on PostgreSQL's CPU and memory resources, which could lead to resource contention under extreme query volumes compared to dedicated, horizontally scalable vector nodes. However, for the expected scale of screening hundreds of candidates per job role, pgvector provides the optimal balance of simplicity, consistency, and query speed. Additionally, co-locating embeddings with candidate metadata simplifies backup and restore operations. Using standard tools like `pg_dump`, the team can backup the entire system state—including relational data and high-dimensional vectors—into a single file. This is far simpler than managing synchronized backups across a relational database and a separate vector database. By keeping the vector index local to PostgreSQL, TalentForge avoids external bandwidth charges and simplifies the provisioning pipeline, proving that co-location of relational data and vector embeddings is the most efficient design for mid-sized enterprise workloads.

## Why Celery and Redis were Chosen for Background Task Orchestration in TalentForge

In TalentForge, resume parsing is a heavy-duty, multi-stage operation involving network calls to Azure AI Document Intelligence, HuggingFace NER, and LLM APIs, averaging 15 to 40 seconds per file. If this ingestion occurred directly within the FastAPI request-response lifecycle, the server's event loop would block, HTTP connections would time out, and recruiters would experience an unresponsive user interface. To decouple these long-running processes, the team chose Celery as the distributed task queue and Redis as the message broker. FastAPI merely validates the uploaded file, saves a placeholder record, and immediately enqueues a `parse_resume_task` to Redis, returning a `202 Accepted` status with a task ID so the frontend can display a progress loader. The actual parsing is executed asynchronously by Celery workers. In the local development environment, Redis runs as a containerized service using the `redis:7-alpine` image, which provides a lightweight and highly performant broker interface. Celery workers run in their own container, communicating with the broker over the bridge network. Celery also provides robust task management, including automatic retries with exponential backoff (`max_retries=3` starting at 2 seconds and doubling) to handle temporary rate limits or network drops from external AI providers. If a task fails repeatedly, it is moved to a dead-letter queue, preventing the worker from getting stuck on corrupted files. The primary tradeoff of using Celery is the operational overhead of managing worker containers and maintaining a message broker, which increases the complexity of local Docker Compose setups. However, this decoupling is a production requirement, ensuring that TalentForge remains responsive and scalable under sudden bulk uploads. Furthermore, using Redis as both a message broker and a caching layer allows the application to store temporary pipeline statuses. FastAPI can query Redis to check the progress of a Celery task, allowing the React frontend to poll for status updates (`uploaded`, `queued`, `parsing`, `parsed`, or `failed`). This prevents the application from making repeated queries to the primary PostgreSQL database just to check task status, reducing database CPU load. By isolating heavy CPU and network-bound task execution to Celery, the core FastAPI server remains free to handle incoming API traffic, ensuring consistent latency for recruiters.

## Why Multi-Model Triangulation was Chosen over Single-LLM Ingestion in TalentForge

In TalentForge, early experiments with single-LLM extraction — sending raw resume text to Gemini with a prompt to extract all fields — produced unacceptably high hallucination rates on structured data like contact details and portfolio URLs. The core problem was that a text-only LLM cannot distinguish between what a document says and what a hyperlink destination contains, nor can it read regex-reliable patterns with the accuracy of a pattern matcher. The triangulation decision replaced a single model with a layered pipeline that assigns each task to the tool best suited to it. Azure Document Intelligence handles layout-aware OCR, which is essential for multi-column resumes where raw text extraction produces column-interleaved garbage. Regex handles contact fields — emails, phone numbers, bare domains — where structured patterns are 100% reliable and hallucination is unacceptable. The HuggingFace `bert-base-NER` model handles named entity extraction at zero per-token cost. Gemini's role is narrowed to synthesis: given clean, independently verified inputs, it maps them onto the Pydantic schema. The architectural tradeoff is increased system complexity and API dependency surface. A failure in any one of Azure DI, HuggingFace, or Gemini introduces a failure mode. This is mitigated by a local fallback parser (PyMuPDF and python-docx) that activates if Azure DI is unavailable, and by Groq serving as the LLM fallback. The hallucination-reduction benefit of triangulation outweighs the orchestration overhead for a recruiting application, where incorrect contact details or fabricated portfolio links would directly damage recruiter trust in TalentForge's output quality.

## Why Deterministic Code was Chosen for Business Logic Calculations in TalentForge

In TalentForge, the principle driving calculation design is a strict separation of concerns: LLMs extract, deterministic code calculates. This boundary was drawn deliberately after observing that generative models are unreliable at arithmetic, date reasoning, and any task where the correct answer must be reproducible across identical inputs. The recruiting domain makes this boundary non-negotiable. A candidate's experience tenure and matched skill count directly determine their ranking score and, by extension, whether a recruiter considers them for a role. If those values are LLM-generated, they are neither auditable nor consistent — the same resume could produce a different score on two different inference runs. Delegating calculations to deterministic Python code in modules like `experience_calculator.py` and `skill_normaliser.py` means the system can produce an exact, step-by-step breakdown of any candidate's score on demand. If a recruiter disputes a result, the calculation is fully traceable. The tradeoff is that deterministic modules require explicit domain knowledge: the skill alias map must be maintained as new technologies emerge, and the date parsing logic must handle the full range of date formats candidates use. These are known, bounded maintenance tasks, whereas LLM non-determinism is an unbounded and invisible risk. This design principle extends to ranking: all scoring computations run inside a single PostgreSQL CTE using SQL arithmetic on indexed columns, not application-level LLM calls. By enforcing that TalentForge's business logic never touches a generative model, the team ensures that every ranking outcome is fair, reproducible, and defensible under recruiter scrutiny.

## Why Serverless Services were Selected for the Future Cloud Deployment of TalentForge

For the future cloud deployment of TalentForge, every infrastructure service was selected against a single primary criterion: scale-to-zero cost behavior. The project's usage pattern is intermittent — recruiters upload resumes in batches, then the system is idle for extended periods. A continuously running server, database, or cache would accrue fixed hourly charges regardless of activity, rapidly exhausting the available student subscription credit. Neon PostgreSQL was chosen as the future managed database because it is the only serverless PostgreSQL provider with native pgvector support, making it a drop-in replacement for the local Docker postgres container at zero migration cost beyond a connection string change. Upstash Redis was chosen over Azure Cache for Redis specifically because the latter costs approximately $16 per month with no idle scale-down, while Upstash charges per command and offers a free tier of 10,000 requests per day — sufficient for a portfolio-scale workload at zero cost. Azure Container Apps was chosen for hosting over virtual machines or Azure Kubernetes Service because it provides managed container orchestration with built-in KEDA integration, removing the operational overhead of cluster management. The Container Apps Jobs execution model is the correct primitive for Celery workers: jobs are triggered by Redis queue depth via KEDA and exist only for the duration of one task execution, meaning zero compute cost between resume uploads. Azure Key Vault was chosen to centralize secrets management rather than using container environment variables, providing audit logs for every secret access and enabling secret rotation without redeployment. Azure Static Web Apps was chosen for the React frontend because it is free for static SPA hosting and generates GitHub Actions deployment workflows and pull request preview URLs automatically. Together these selections enable TalentForge to maintain a projected monthly cost under $25 while preserving full enterprise-grade operational capabilities.
