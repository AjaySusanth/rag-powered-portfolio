# ClassSync (Automated Classwork) - Technical Decisions Log

## Why n8n was Chosen for Workflow Orchestration in ClassSync
In the ClassSync (Automated Classwork) project, n8n was chosen as the workflow automation and orchestration engine instead of implementing inline schedulers (like node-cron) or managing worker queues (like BullMQ) inside the Node.js application. Inline cron schedulers run within the primary application process, meaning that intensive cron executions or database polling can block the event loop and degrade the responsiveness of client-facing REST endpoints. Managing worker queues introduces additional infrastructure requirements (such as a Redis instance) and requires writing significant boilerplate code for job retries, message formatting, and third-party API integration. By offloading these responsibilities to n8n, the ClassSync API remains lightweight, stateless, and focused purely on CRUD operations and authorization rules. Furthermore, n8n provides a visual interface for constructing pipelines, built-in nodes for Telegram operations, and automatic retry behaviors. This allows the development team to adjust notification timing, update chat layouts, or add error-handling flows without rewriting and redeploying the core application code. The trade-off is the operational overhead of hosting and maintaining a separate n8n instance, but this cost is offset by faster development velocity and isolation of background scheduling loads from the main server process.

## Why PostgreSQL with Prisma ORM was Chosen for ClassSync
In the ClassSync (Automated Classwork) project, a relational schema on PostgreSQL 16 was selected instead of a NoSQL database like MongoDB. Relational databases are critical for educational platforms because the system requirements involve strict integrity constraints and cascading operations. For example, student classroom memberships, assignments, submissions, and reminders are highly interconnected. Using a relational database ensures that deleting an assignment automatically handles corresponding submissions and reminders through foreign key constraints and transactional playbooks. To interface with the database, Prisma ORM v7 was chosen. Prisma provides type-safe query generation, which eliminates runtime database type errors and speeds up feature development. It also automates database schema migrations, ensuring that local development and production environments remain synchronized. While a document database like MongoDB could offer flexible schemas and high write throughput, it would require application-level validation to prevent orphaned records, which increases the likelihood of data consistency bugs. The structured nature of PostgreSQL, combined with Prisma's transaction API, guarantees that complex enrollments and submission updates succeed atomically or fail completely.

## Why a Channel-Agnostic Notification Service Abstraction was Chosen for ClassSync
In the ClassSync (Automated Classwork) project, notification delivery is built around a channel-agnostic registry pattern rather than being tightly coupled to the Telegram Bot API. Educational platforms must remain adaptable to changing communication channels; for instance, students might prefer WhatsApp or email notifications over Telegram. If the application controller code directly invoked the Telegram API, adding support for a new channel would require modifying every controller that sends notifications, creating a high maintenance burden and increasing the risk of regression bugs. To address this, the codebase implements a base `NotificationChannel` class that defines a common interface for dispatching messages. A concrete `TelegramChannel` class inherits from this base and handles Telegram-specific API calls. The `NotificationService` acts as a central registry where channels are registered. When a notification needs to be sent, the controller simply calls the service, which dispatches the message via the active channel and logs the result to the `NotificationLog` table. This design ensures that adding new notification channels requires only implementing the abstraction and registering the new channel, leaving the core business logic untouched.

## Why an Abstracted Storage Provider Architecture was Chosen for ClassSync
In the ClassSync (Automated Classwork) project, document storage is abstracted through a provider pattern rather than relying on local disk storage or direct calls to a specific cloud vendor. Storing uploaded assignments and student submissions on the server's local file system would make the application stateful. This statefulness would prevent deploying the application on serverless platforms (like Azure Container Apps) that scale to zero, as files would be lost when container instances are recycled. To resolve this, the system implements a `StorageProvider` interface that abstracts file uploads and deletions. A concrete `SupabaseStorage` class implements this interface, using the `@supabase/supabase-js` SDK to upload files to a remote object storage bucket and retrieve public URLs. The controllers handle file buffers and interact only with the `StorageProvider` contract. This abstraction ensures that migrating from Supabase Storage to alternative solutions like AWS S3 or Azure Blob Storage only requires writing a new class implementing the interface, ensuring the upload and grading controllers remain unaffected by cloud infrastructure changes.

## Why Service-to-Service Authentication via Static API Key was Chosen for ClassSync
In the ClassSync (Automated Classwork) project, a static API key (configured as `INTERNAL_API_KEY`) is used to authenticate internal calls from n8n instead of standard JSON Web Tokens (JWT). The backend endpoints are split into two categories: user-facing routes and service-to-service internal routes. User-facing routes require JWTs, which are generated upon registration or login and represent the security context of a specific student or teacher. However, n8n workflows run asynchronously in the background and do not operate under a specific user's session. Generating short-lived JWTs for n8n would require creating a system user account or writing complex token-refresh logic within n8n workflows. To secure endpoints like `/api/internal/reminders/due-soon` and `/api/internal/telegram-linked` without adding unnecessary complexity, the `serviceAuth` middleware validates a static header token. This static key is stored securely in environment variables on both the Express server and the n8n instance, ensuring that only authorized automation workflows can access internal administrative endpoints while preventing public access to enrollment and reminder schedules.
