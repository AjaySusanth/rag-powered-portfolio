# TalentForge Architecture Overview

## High-Level System Topology and Component Interaction in TalentForge

In the TalentForge architecture, the system is designed as a modular monolith with an asynchronous back-end processing queue and a front-end single-page application. The front-end is built using React 19, TypeScript, and Vite, serving as the user interface for recruiters to create job roles, configure scoring templates, upload resumes, and view candidates sorted by dynamic matching scores. Communication between the React front-end and the back-end is managed via asynchronous RESTful APIs exposed by a FastAPI application server running Python 3.11+. The FastAPI server acts as the central coordinator, validating API requests, managing rate limits, interacting with the primary database, and coordinating file storage operations. For object storage, the system integrates with Supabase Storage (an S3-compatible service) to securely persist uploaded resume files (PDF, DOCX, and images) and retrieve them via secure presigned URLs, avoiding proxying file bytes directly through the application server. The back-end relies on a PostgreSQL database equipped with the pgvector extension for local vector operations and semantic data persistence, keeping candidate metadata co-located with their vector representations. To prevent long-running tasks from blocking the FastAPI event loop, an asynchronous background task queue is implemented using Celery, with Redis serving as the message broker and caching layer. When a recruiter uploads resumes, the FastAPI server records the upload event and enqueues a parsing task in Redis. A Celery worker pool, which runs independently, retrieves these tasks from Redis and executes the heavy-duty extraction, embedding, and normalization pipeline. After processing is complete, the Celery workers save the extracted profiles and vectors directly to the PostgreSQL database and update the upload status, which the front-end polls to provide live updates. This component interaction pattern ensures that the user experience remains responsive, decoupling the slow 15-to-40-second resume processing time from the synchronous HTTP request-response cycle. To secure these interactions, the FastAPI backend in TalentForge enforces API key authentication using custom middleware that validates the `x-api-key` header on all protected routes. Additionally, the backend uses `slowapi` to implement rate-limiting rules, such as limiting resume uploads to 10 per minute and ranking requests to 20 per minute, preventing abuse of downstream AI services. For operational visibility, the backend integrates structured JSON logging using `structlog`, which attaches a unique `request_id` to every log message, enabling recruiters or administrators to trace a request from the initial API call down to the background Celery task execution. This distributed tracing capability is crucial for identifying latency bottlenecks in the multi-stage extraction pipeline.

## Asynchronous Resume Ingestion and Triangulation Pipeline in TalentForge

In TalentForge, the resume processing pipeline operates as a stateful, twelve-step asynchronous workflow designed to extract high-fidelity structured profiles from raw documents. The ingestion begins when a recruiter uploads files, which are validated for MIME type using `python-magic` to prevent malicious extensions from executing on backend services. Once stored in Supabase, a task is enqueued to Celery. The Celery worker downloads the document bytes and initiates a pre-extraction stage. To avoid relying on LLM heuristics to guess URLs from displayed text, TalentForge uses python libraries (`fitz` for PDFs and `zipfile` for DOCX files) to read the underlying document structure, extracting authoritative hyperlink destinations (such as LinkedIn profiles or GitHub repositories) directly from hyperlink annotations or Word document relationship XML files. Next, the document text is processed. The primary OCR and layout engine is Azure AI Document Intelligence using the `prebuilt-layout` model, which excels at parsing complex multi-column resumes, tables, and skills grids in reading order, rather than using raw text storage order which would interleave columns. In parallel, a deterministic regex engine runs to identify emails, phone numbers, and bare-domain portfolios. To guide the subsequent language model step, the worker calls the HuggingFace Inference API to execute a lightweight `bert-base-NER` Named Entity Recognition model, extracting names, locations, and organizations. The raw text, regex outputs, and NER hints are then injected as ground-truth context into `gemini-2.0-flash`, which synthesizes the data into a validated JSON structure mapping to a strict Pydantic schema. Once synthesized, the pipeline performs deterministic canonicalization: a custom interval-merge algorithm parses work dates, resolves overlaps, and calculates true tenure, while a tiered skill normalizer maps skills to canonical names using an alias dictionary, falling back to a Groq-hosted Llama 3.3 70B model for unknown skills. Finally, the candidate's structured profile is summarized and converted into a 1536-dimensional semantic vector using the OpenAI `text-embedding-3-small` model, which is persisted along with the JSON profile in PostgreSQL, updating the upload status to parsed so it is ready for ranking.

## Hybrid Candidate Ranking and SQL Schema in TalentForge

To rank candidates against job descriptions quickly and accurately, TalentForge implements a hybrid ranking engine executed as a single, high-performance database-level Common Table Expression (CTE) query in PostgreSQL. This in-database scoring model eliminates the need to load candidate profiles into application memory, achieving sub-100 millisecond execution times across hundreds of candidates. The hybrid score is calculated by combining a semantic score (accounting for 40% of the total) with a rule-based score (accounting for 60% of the total). The semantic score is derived by computing the cosine similarity between the 1536-dimensional embedding of the Job Description and the Candidate's profile vector, using pgvector's cosine distance operator `1 - (c.embedding <=> jd.embedding)`. To optimize these vector comparisons, the database features a Hierarchical Navigable Small World (HNSW) index on the candidates' embedding column. The remaining 60% of the score is determined by rule-based business logic, which evaluates candidate profiles against specific requirements defined in the Job Description, such as matching skills, experience tenure, practical projects, professional roles, and certifications. Recruiters can adjust these sub-weights using a frontend slider interface, and the database enforces that they sum to exactly 100% via a check constraint (`weights_sum_100`) on the `job_scoring_config` table. To protect against unqualified candidates, TalentForge applies a mandatory gate multiplier: if a candidate matches less than 40% of the mandatory skills specified in the job description, their rule-based score is heavily penalized, automatically pushing them down the rankings. Additionally, for the top 5 ranked candidates, the ranking service enqueues an asynchronous task to generate a natural language fit justification. This task uses `gemini-2.0-flash` (or a Groq-hosted `llama-3.1-8b-instant` fallback) to write exactly two factual sentences explaining why the candidate matches or falls short, caching the text in the `job_matches` table. This combination of semantic vector search, customizable rule weights, and automated AI justifications provides recruiters with immediate, auditable insights into candidate suitability.

## Local Containerized Orchestration and Development Topology in TalentForge

In TalentForge, the local development and testing environment is fully containerized and orchestrated using Docker Compose. The services defined in the configuration include `postgres` running PostgreSQL 16 with the pgvector extension, `redis` running Redis 7, the `api` service running the FastAPI application, the `worker` service running the Celery background worker, and the `frontend` running React 19. Rather than deploying directly to the cloud at this stage, the codebase uses a local orchestration topology where all services communicate over a shared Docker bridge network. The `postgres` container maps its internal port 5432 to the host machine and persists data across container restarts using a named Docker volume called `pgdata`. The `redis` container maps port 6379 to the host, acting as the message broker for background tasks. The `api` and `worker` containers are both built from the same multi-stage Dockerfile located in the backend folder, using the `development` target which mounts the local backend source directory. This allows for live code reloading during development. The API container runs the Uvicorn server, while the worker container starts the Celery worker process. Both read credentials and API endpoints from a local `.env` file via environment variables. The `frontend` service builds the React single-page app and maps port 3000 to the host, forwarding API requests to `http://localhost:8000/api/v1` via standard environment variables. This containerized design mirrors the production layout locally, allowing developers to run, test, and debug the entire resume processing and ranking pipeline on a single machine without modifying system dependencies. To coordinate startup order, Docker Compose uses health checks. The `api` and `worker` services both depend on `postgres` and `redis` being healthy. The database container defines a healthcheck that runs `pg_isready -U postgres`, while the Redis container uses `redis-cli ping`. This prevents the application and worker containers from launching before their backing stores are ready, avoiding connection errors on startup. By utilizing local volume mounts and network bridges, TalentForge provides a self-contained, reproducible development stack that acts as a robust staging ground before any production cloud deployment.

## Future Cloud Deployment Topology for TalentForge

Although TalentForge currently runs locally via Docker Compose, the target production architecture is a fully serverless topology on Microsoft Azure. The component layout maps directly onto the current local stack, with each Docker service replaced by its cloud equivalent. The React SPA moves to Azure Static Web Apps, which provides a global CDN, automatic HTTPS, and isolated pull request preview environments. The FastAPI backend moves to Azure Container Apps, configured with HTTP-based autoscaling from zero to three replicas. The Celery worker pool moves to Azure Container Apps Jobs, a dedicated execution primitive for event-driven background workers. The local Redis broker is replaced by Upstash Redis, and the local PostgreSQL container is replaced by Neon PostgreSQL, both of which retain the same connection protocol used locally. KEDA (Kubernetes Event-driven Autoscaling) sits between Upstash and the Container Apps Jobs runtime: it watches the length of the default Celery queue in Upstash Redis and provisions one worker job replica per pending task, scaling back to zero when the queue is empty. The Neon database retains the same schema as the local container, including the pgvector extension, the HNSW index on the `candidates.embedding` column, and all Alembic migrations. Azure Key Vault replaces local `.env` file secrets, with managed identities granting the Container Apps runtime read access at startup. Azure Application Insights collects structured logs and distributed traces from both the FastAPI backend and the Celery workers, correlated using a shared `operation_id`. The CI/CD pipeline runs on GitHub Actions with OIDC authentication to Azure, building separate Docker images for the API and the worker, pushing them to Azure Container Registry, and deploying each service independently. A post-deployment AI validation gate uploads a fixture resume to the staging environment and asserts that the full parsing pipeline produces a valid, embedded candidate record before promoting to production. This topology preserves all of TalentForge's local architectural properties — asynchronous task decoupling, co-located vector storage, hybrid SQL ranking — while adding managed scalability and observability.
