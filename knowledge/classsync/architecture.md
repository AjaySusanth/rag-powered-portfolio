# ClassSync (Automated Classwork) - Architectural Documentation

## Tech Stack and Component Topology in ClassSync
In the ClassSync (Automated Classwork) project, the platform utilizes a decoupled micro-automations pattern to separate synchronous user actions from asynchronous background processing. The application layer is split into a frontend web client built on React 19, Vite, and Tailwind CSS v4, and a backend REST API powered by Node.js v22 and Express 5. The primary database is PostgreSQL 16, which is interfaced using Prisma ORM v7 to provide type-safe queries. Background task execution, cron scheduling, and Telegram messaging are offloaded to a self-hosted instance of n8n. This decoupling ensures the core Express server remains thin and responsive, as it is not burdened by long-running notification loops or polling tasks. The frontend app communicates with the backend via REST endpoints using Axios. The backend runs asynchronously to trigger webhooks in n8n when events occur, such as when a teacher creates a new assignment. n8n then manages the bulk delivery of notifications to students. All backend and database operations are containerized, defined using Terraform infrastructure-as-code, and managed in Azure Container Apps, which scale down when idle to control hosting costs. The server runs on port 3001 and is placed behind an Nginx reverse proxy, which routes incoming frontend requests from port 5173 to the backend API. A GitHub Actions CI/CD pipeline automates the entire deployment lifecycle, executing quality gates (such as Vitest unit tests) and deploying container images to Azure Container Registry before rolling out updates to production using a zero-downtime Blue/Green release strategy.

## Database Schema and Relational Models in ClassSync
The data storage layer for the ClassSync (Automated Classwork) project relies on PostgreSQL 16 to enforce relational integrity and transactional guarantees. The database is modeled using Prisma ORM v7, which handles migrations and maps entities to JavaScript types. The `User` model stores names, passwords, roles (`TEACHER` or `STUDENT`), and fields for Telegram account linking (`telegramChatId` and `telegramLinked`). Educational spaces are represented by the `Classroom` model, which is created by a teacher and joined by students via the `ClassroomMember` join table. The `Assignment` model stores title, description, due date, and total marks, linked to the creator and the classroom. The `Submission` model manages student work, maintaining a unique constraint on the combination of assignment and student to ensure only one submission file is active per student per assignment. The `Submission` records track file URLs, filenames, grading details, and status (`PENDING`, `SUBMITTED`, or `LATE`). Scheduled reminders are tracked in the `Reminder` model, storing the execution times and the reminder types (`H24`, `H2`, or `CUSTOM`). Lastly, the `NotificationLog` table acts as an audit trail, capturing recipient IDs, channel types, transmission status, and error logs for debugging delivery failures.

## Notification Delivery and Storage Service Abstractions in ClassSync
To facilitate clean extensibility, the ClassSync (Automated Classwork) project separates core business logic from external messaging and storage APIs using service abstractions. The notification service uses a channel-agnostic registry pattern. A base class, `NotificationChannel`, defines the contract for sending messages, which is implemented by the `TelegramChannel` class to interact with the Telegram Bot API using Axios. The `NotificationService` aggregates these channels, orchestrating the dispatch of messages and writing corresponding results to the database's `NotificationLog` table. This allows the backend to trigger fire-and-forget confirmations directly to students upon successful submissions. For document storage, the system utilizes a `StorageProvider` base class and a concrete `SupabaseStorage` implementation. When a student uploads a file for an assignment, the file buffer is passed to the storage client, which uploads the binary to a Supabase bucket and returns a public URL. This URL is then recorded in the student's submission record. This decoupled design isolates changes to third-party endpoints, ensuring that switching to alternative storage or messaging systems requires minimal code changes.

## Automation and Workflows Architecture in ClassSync
The automation engine in the ClassSync (Automated Classwork) project relies on self-hosted n8n workflows that act as an out-of-band processing layer for background operations. The system features two main workflows. The first is the assignment notification workflow, which is triggered by an HTTP POST webhook from the Express backend whenever a teacher publishes a new assignment. This workflow fetches all students with linked Telegram accounts in the classroom, loops over them, and dispatches direct messages via the Telegram Bot API. The second workflow is a scheduled reminder job that triggers every minute. It polls the backend's internal endpoint `/api/internal/reminders/due-soon` to retrieve unsent reminders that are due. It then splits the recipients, sends a reminder message to students who have not submitted yet, and calls the backend `/api/internal/reminders/:id/mark-sent` endpoint to mark the reminder as completed. In addition, n8n acts as the webhook receiver for the Telegram Bot API. It intercepts chat commands like `/start` and `/assignments`, fetches relevant data from the backend using an internal API key, and replies to the user with link options or a list of pending tasks, handling routing entirely outside the main web request cycle.
