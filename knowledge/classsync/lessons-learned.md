# ClassSync (Automated Classwork) - Lessons Learned

## Value of Mocking Database Adapters in Unit Testing in ClassSync
In the ClassSync (Automated Classwork) project, implementing unit tests in the `health.test.js` file demonstrated the benefits of mocking the database adapter during local development and continuous integration. Initially, verifying backend routing and validation logic required active database connections, which meant tests were slow, prone to network timeouts, and difficult to configure in automated environments. By utilizing Vitest's mocking tools, the Prisma client, PostgreSQL connection pool, and PG adapters were completely mocked out. This allowed the test suite to verify route middleware, authorization guards, and validation responses in isolation. Mocking database adapters ensures that the test runner executes in milliseconds and remains immune to database connection failures. The lesson learned is that for core controller and middleware verification, complete separation from physical database instances should be established early, as it increases pipeline reliability and decreases development latency.

## Fire-and-Forget Pattern for Egress Channels in ClassSync
In the ClassSync (Automated Classwork) project, dispatching Telegram confirmation messages synchronously during student file submissions introduced latency and potential failure points. Originally, when a student submitted an assignment, the API controller waited for the Telegram API to send a success response before responding to the student's upload. If the Telegram Bot API experienced high latency, or if the student's chat ID was blocked, the submission request would hang or fail, causing the client upload to timeout. To resolve this, the system was updated to process Telegram confirmations using a fire-and-forget asynchronous immediately invoked function expression (IIFE). This pattern returns the upload status to the student immediately, while the Telegram message is dispatched in the background. If the background notification fails, the error is logged without disrupting the student's transaction. The takeaway is that external egress channels should be isolated from the primary request cycle to protect the reliability and performance of core user operations.

## Automated Self-Healing of Scheduled Tasks in ClassSync
In the ClassSync (Automated Classwork) project, scheduled workflows that poll for due reminders were prone to clogging. If a teacher deleted an assignment, or if all students had submitted their work early, the scheduler would continue to fetch empty reminder records, wasting database queries and creating unnecessary logs. To fix this, a self-healing mechanism was added to the `/api/internal/reminders/due-soon` endpoint. When the server processes a request for due reminders, it automatically checks if any active students remain pending. If all students have completed their submissions or if no eligible recipients are found, the server automatically updates the reminder's `sent` status to `true` in a bulk operation. This self-healing design prevents redundant polling, reduces logging noise, and ensures that the scheduling engine only focuses on active reminders, increasing overall system efficiency.

## Consolidating Webhook Triggers in Third-Party APIs in ClassSync
In the ClassSync (Automated Classwork) project, working with the Telegram Bot API showed the necessity of consolidating incoming webhooks when using third-party APIs. Attempting to deploy multiple, independent n8n workflows that each registered their own webhook URLs with the same Telegram bot token resulted in race conditions where each registration would overwrite the previous one. This made it impossible to run the account-linking workflow and the assignments command workflow at the same time. The lesson learned is that when integrating bots with strict single-webhook constraints, all incoming traffic must be funneled through a single routing controller. By merging the flows into a single "Telegram Command Router" in n8n, the system can parse the message payload and route it dynamically, avoiding configuration conflicts and streamlining update management.
