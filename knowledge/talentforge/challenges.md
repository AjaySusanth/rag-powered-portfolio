# TalentForge Development Challenges

## Resolving Column Interleaving in Two-Column Resumes in TalentForge

In TalentForge, a significant technical hurdle was encountered during the extraction of text from two-column resumes. Many candidates format their resumes with a left-hand column for contact details and skills, and a right-hand column for professional experience. When using Azure Document Intelligence, the SDK provides a `.content` property that returns the entire extracted text as a single string. However, because `.content` reads characters in the file's underlying XML layout storage order, it interleaved the text from the two columns. This produced garbled text where skills and company names were mashed together, making it impossible for the downstream Gemini model to extract structured data or associate descriptions with the correct job titles. To resolve this issue, the engineering team modified the Azure client to bypass `result.content` entirely. Instead, the parser iterates through `result.paragraphs`. Azure Document Intelligence's layout engine automatically analyzes the visual structure of the document and sorts paragraphs in reading order, rather than file-storage order. The team implemented a paragraph crawler that filters out headers and footers (which polluted the token space) and joins the clean paragraph blocks with newlines. For cases where the pipeline falls back to PyMuPDF due to Azure API issues, a similar column interleaving problem occurred. The team resolved this by extracting text in 'blocks' mode (which provides coordinates for each text chunk) and sorting the blocks by vertical band (rounding the y-coordinate to 20-pixel rows) and then horizontally. This visual sorting approximation successfully restored the correct reading order for two-column layouts, ensuring that the system captures candidate data accurately across both primary and fallback parsing paths. This layout resolution was verified by uploading a set of highly stylized, multi-column sample resumes and verifying the Celery worker logs. By shifting to paragraph-level reading order, the system reduced token waste and eliminated extraction failures caused by interleaved strings, proving that layout-aware text extraction is critical for maintaining high parsing accuracy on real-world resume formats. Additionally, the team added table-specific logic: because Azure DI returns table cells with row and column indices separated from normal paragraphs, the parser reconstructs table data row-by-row into a pipe-delimited format. This ensures that skills grids and education details formatted as tables are not dropped, preserving data integrity.

## Resolving Inaccurate Experience Math and Overlapping Roles in TalentForge

In TalentForge, calculating a candidate's total years of experience was a persistent failure point in early versions of the pipeline. The initial approach delegated date arithmetic directly to Gemini: the model was asked to extract work history and output the total tenure as a single number. In practice, the LLM consistently produced inflated totals. It added the durations of overlapping roles together (e.g., a candidate holding a full-time job and a concurrent part-time freelance contract would have both counted in full), and it ignored career gaps, treating the span from the first employment start date to the present as one continuous block. The symptom was clear: a candidate with a five-year working window could be reported as having eight or nine years of experience. To fix this, the developer decoupled date arithmetic from the LLM entirely. The model's scope was narrowed to a single, well-defined task: extract each role's start and end dates in a structured `YYYY-MM` format, with `Present` for ongoing roles. Calculation was delegated to a deterministic `experience_calculator.py` module. The module converts every date string into a linear month index (`year * 12 + month`), defaulting open-ended roles to the current month. It then sorts all intervals chronologically and applies an interval-merge pass: if two intervals overlap or are adjacent (start of next ≤ end of previous + 1), they are merged into a single span. The total experience is the sum of all merged interval lengths divided by twelve. The module also performs a date-sanity check on input — if a role's end date precedes its start date, the values are automatically swapped and a warning is logged. After this fix, experience scores in TalentForge became fully reproducible and matched recruiter expectations across all test resumes, including profiles with internships, career breaks, and concurrent freelance work.

## Extracting Hidden Hyperlinks from Resume Files in TalentForge

In TalentForge, candidate portfolio links (LinkedIn profiles, GitHub repositories, personal websites) were frequently missing from extracted profiles despite being clearly present on the uploaded resume. The root cause was identified during investigation: candidates rarely write out raw URLs in resume text. Instead, they use anchor text — words like "GitHub", "Portfolio", or a company logo — which hyperlinks to the actual URL destination. Azure Document Intelligence's `prebuilt-layout` model processes visual content only: it reads the display text but silently discards the link annotation that holds the destination URL. The result was candidate records with empty `github_url`, `linkedin_url`, and `portfolio` fields even though those links existed in the uploaded file. The fix was a dedicated pre-extraction stage in `hyperlink_extractor.py` that runs before the document is sent to Azure DI. For PDFs, the module uses PyMuPDF (`fitz`) to iterate each page's link dictionary and retrieve the raw `uri` field from each link annotation — the binary metadata that OCR never sees. For DOCX files, which are ZIP archives containing XML files, the module opens `word/_rels/document.xml.rels` and parses all `<Relationship>` entries with `TargetMode="External"` to collect every outbound hyperlink. The harvested URLs are then filtered and classified by pattern: LinkedIn profile URLs, GitHub user and repository URLs, and generic project links are separated into named buckets and deduplicated. These structured hyperlink results are injected as ground-truth hints into the Gemini prompt before synthesis, giving the model verified URL destinations to assign to the correct profile fields rather than asking it to reconstruct them from display text. After this fix, portfolio and social profile fields in TalentForge were populated reliably across all tested resume formats, eliminating a significant gap in candidate data quality.

## Resolving Wrong Content Type and HTTP 400 Errors in Azure DI Requests in TalentForge

In TalentForge, a critical bug emerged during the integration of Azure Document Intelligence that caused the resume parsing pipeline to crash with HTTP 400 Bad Request errors on every upload. The issue was traced to `azure_di_client.py` inside the `extract_text` function. The code was configured to send the file bytes to Azure's `begin_analyze_document` endpoint, passing the document's actual MIME type (e.g., `application/pdf` or `image/png`) to the `content_type` parameter of the SDK. While this seemed logical, it conflicted with how the Azure Python SDK handles file transmissions. When using the `bytes_source` parameter, the SDK wraps the binary data in a JSON envelope before sending it. Passing the resume's MIME type to the `content_type` parameter forced the SDK to set the HTTP Content-Type header of the request to `application/pdf`, while the actual body of the request was a JSON string. This header mismatch caused Azure's API gateway to reject the request immediately as malformed. To resolve this, the developer updated the client to hardcode the `content_type` parameter to `application/json` whenever `bytes_source` is used. Azure Document Intelligence automatically infers the underlying file type (whether it is a PDF, DOCX, or JPEG) directly from the binary stream itself, rendering the custom MIME type header unnecessary for decoding. After applying this fix, the SDK correctly transmitted the JSON envelope with matching headers, and the API successfully processed the documents. This fix was verified by checking the Celery worker logs to confirm that the `azure_di_extraction_complete` event was logged with page counts greater than zero, ensuring that the ingestion pipeline could reliably communicate with Azure services. This bug highlighted the importance of understanding SDK abstraction layers. The developer added a comprehensive unit test in the local test suite to mock the Azure DI client and verify that the outgoing payload headers are always set to `application/json` when analyzing bytes. This safeguard prevents future regression bugs if the SDK is upgraded, ensuring long-term stability for the ingestion pipeline.
