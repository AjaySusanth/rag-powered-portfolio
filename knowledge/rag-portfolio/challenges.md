# RAG-Powered Developer Portfolio - Technical Challenges

## Resolving Context Starvation and Duplicate Domination in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, a significant challenge emerged where dense vector search returned multiple highly similar chunks from a single document. For example, a query about AKS deployment would retrieve five different sections of `n8n-aks-platform/lessons-learned.md`, crowding out other crucial files like Terraform modules or Helm charts. This lowered the diversity of the context window and starved the LLM of specific code configurations. To resolve this, a Source Diversification post-processing step was added after Reciprocal Rank Fusion. This component groups retrieved chunks by source file path and retains only the top-ranked chunk per file up to a configurable limit (default: 3). In offline testing against the AKS deployment dataset, this increased the Average Unique Sources@5 from 4.40 to 5.00, allowing specific code templates to rise into the top-K and improving MRR from 0.456 to 0.467.

## Managing Split Boundaries and Chunk Quality in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, standard recursive character text splitters would cut through the middle of code blocks, lists, or tables. This caused loss of structural context and resulted in poor embeddings and RAG answers. Additionally, manual documents contained internal markdown comment blocks (`<!-- ... -->`) detailing RAG formatting decisions, which wasted token budget and polluted embeddings. To fix this, a heading-aware two-pass chunker was built. The first pass splits files at Markdown headers (#/##/###), ensuring sections remain intact. The second pass applies a token-aware sliding window within each section, prepending the parent heading to every chunk. All comments are stripped via regular expressions before tokenization. Very small sections are folded into the previous chunk to prevent near-empty embeddings.

## Mitigating Latency in the Multi-Pass LLM Retrieval Pipeline in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, executing query rewriting, parallel vector and lexical searches, RRF, retrieval grading, and citation attribution sequentially introduced high end-to-end latency. Standard HTTP REST requests would hang for several seconds. To solve this, three optimizations were implemented: parallel execution of vector database queries and BM25 indexing using `asyncio.gather`, a single-pass Query Rewriter that decides if rewriting is necessary, and Server-Sent Events (SSE) via the FastAPI `/chat` endpoint to stream tokens to the frontend in real time. Citation attribution was also deferred to a separate asynchronous pass once generation completed, keeping the initial token latency under 3 seconds.
