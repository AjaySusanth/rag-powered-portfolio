# ClassSync (Automated Classwork) - Technical Challenges

## Handling Telegram Webhook Conflicts and Single-Webhook Constraint in ClassSync
In the ClassSync (Automated Classwork) project, integrating the Telegram Bot API presented a significant bottleneck because Telegram restricts each bot to a single webhook URL. The initial development plan involved setting up separate n8n workflows with their own webhook endpoints: one to handle account linking when a student ran `/start`, and another to handle fetching and displaying assignments when a student ran `/assignments`. However, setting the webhook URL in one workflow automatically deactivated the webhook for the other workflow, leading to random connection drops and broken commands. To overcome this limitation, the system architecture was refactored to use a single, unified "Telegram Command Router" workflow in n8n. In this router, all incoming bot updates are sent to a single n8n trigger. The workflow parses the incoming text message, extracts the command name, and uses a Switch node to route the request to corresponding branches (such as account linking, fetching assignments, or displaying help messages). This consolidation resolved the webhook conflict, simplified the bot architecture, and provided a single entry point for debugging incoming messages.

## Student Enrollments and Missing Past Assignment Records in ClassSync
In the ClassSync (Automated Classwork) project, a major data consistency issue emerged when students joined a classroom mid-course using an invite code. Initially, when a teacher created a new assignment, the system generated empty `Submission` records with a `PENDING` status for all students currently enrolled in the classroom. However, if a new student joined the classroom later, they had no existing `Submission` records for previously created assignments. As a result, past assignments were invisible on their student dashboard, and the system would fail to track their submissions or send them overdue reminders. To resolve this problem, the classroom enrollment logic in `classroomController.js` was refactored. The `joinClassroom` controller now executes a multi-step database operation inside a Prisma transaction. When a student enters an invite code, the transaction first creates the `ClassroomMember` enrollment record. Next, it queries all existing assignments for that classroom. Finally, it uses a bulk insert (`createMany` with a `skipDuplicates` flag) to backfill `PENDING` submission placeholders for the student for all past assignments. By wrapping these operations in a transaction, the system ensures that a student is never partially enrolled, preventing orphaned records and ensuring dashboard consistency.

## Preventing Duplicate and Conflicting Telegram Account Links in ClassSync
In the ClassSync (Automated Classwork) project, managing Telegram account linking introduced the risk of data conflicts where multiple users could try to link to the same Telegram chat ID, or a user could attempt to link to multiple chats. This could result in incorrect notification routing or duplicate messages. To prevent these conflicts, the system uses unique constraints and validation checks. The `User` model defines the `telegramChatId` field with a `@unique` constraint in the database schema. In the `telegramLinkController.js`, the `linkTelegramChat` controller verifies that the target chat ID is not already linked to another user. If it is, the server rejects the request with a `409 Conflict` error. The controller also checks if the user is already linked to a different chat ID before proceeding. In the n8n router workflow, these API errors are caught, and the bot replies to the student with clear, user-friendly error messages (e.g., "⚠️ This Telegram account is already linked to another user."). This ensures that account linkages remain strictly one-to-one and prevents routing errors in the notification pipeline.

## Mitigating Race Conditions on Graded Submission Updates in ClassSync
In the ClassSync (Automated Classwork) project, handling student submissions required careful concurrency control to prevent students from overwriting files that had already been graded or resubmitting work after the deadline. If a student uploaded a new file while the teacher was grading the previous submission, it could result in outdated grades or lost feedback. To resolve this concurrency challenge, the `submitAssignment` controller in `submissionController.js` applies strict state checks and database locks. When a student uploads a file, the controller first queries the existing submission record. If the submission has already been graded (i.e., `grade` is not null) or if the current time exceeds the assignment's due date, the request is immediately rejected with a `403 Forbidden` error. Furthermore, to prevent race conditions during updates, the database update is executed using a Prisma `updateMany` statement with a conditional query that checks if the `grade` remains null. If the update matches zero rows (indicating that the submission was graded between the initial query and the update execution), the system deletes the newly uploaded file from Supabase Storage and returns a locking error to the client, preserving the integrity of the graded work.
