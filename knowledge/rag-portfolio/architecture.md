# RAG-Powered Developer Portfolio - Architectural Documentation

## Tech Stack and Component Topology in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, the application architecture is designed as a decoupled full-stack system consisting of a client-side user interface and a server-side REST API. The frontend is built using Next.js 16 (App Router), React 19, and TypeScript, utilizing Tailwind CSS for styling and custom theme support. The client application is hosted on Vercel and communicates with the backend via REST endpoints. For streaming chat answers in real-time, the frontend uses Server-Sent Events (SSE) to render tokens as they are generated. The backend API is built with FastAPI running on Python 3.12 and Uvicorn, containerized using Docker, and deployed on Render. The database layer utilizes serverless Neon PostgreSQL 16 with the pgvector extension for high-performance vector search, and a serverless Upstash Redis 7 cache. Local development uses Docker Compose to run PostgreSQL and Redis alpine images. For all LLM operations—including answer generation, query rewriting, retrieval grading, and citation attribution—the application is integrated with the Google Gemini API (defaulting to the highly cost-effective and quota-generous gemini-3.1-flash-lite model), with Groq llama-3.1-70b serving as the fallback provider.

## Database Schema and pgvector Storage in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, document chunks and their high-dimensional vector embeddings are stored in a PostgreSQL database using pgvector. The database schema consists of a dedicated table for document segments. Each record holds an autogenerated UUID `id`, a string representing the project namespace (such as `talentforge`, `classsync`, `n8n-aks-platform`, or `__global__` for global identity documents), a document layer category (`identity`, `design`, or `artifact`), a file path identifier `source_file`, a Markdown heading string `heading` denoting the parent section, an integer `chunk_index` indicating position, the raw text content string `content`, and a 1536-dimensional float vector column named `embedding`. To optimize semantic search query speeds, the system configures a Hierarchical Navigable Small World (HNSW) index on the embedding vector column. This index uses cosine similarity operator class (`vector_cosine_ops`), which measures distance between embeddings generated by the Azure OpenAI `text-embedding-3-small` model. This structure ensures that vector searches are fast and memory-efficient.

## Hybrid Retrieval and Reciprocal Rank Fusion in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, a hybrid retrieval pipeline is implemented to merge semantic vector matches and lexical keyword matches. When a query is received, the system queries the pgvector HNSW database and the `rank-bm25` index in parallel. The results of both searches are merged using Reciprocal Rank Fusion (RRF), which ranks documents without requiring score normalization. The RRF algorithm uses a constant parameter value of `k=60`, meaning each document's fused score is computed as `1 / (60 + rank)`. Following fusion, the pipeline applies a Source Diversification filter. This stage groups the top-fused chunks by their source path and limits the number of chunks allowed from any single file (governed by the `DIVERSIFICATION_MAX_PER_SOURCE` configuration, which defaults to 3). This prevents a single lengthy document from dominating the context window and ensures a diverse range of context files.

## Self-Healing and Query Preprocessing in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, every incoming query passes through a multi-stage preprocessing sequence to protect backend APIs and improve retrieval accuracy. First, the query is analyzed by the Prompt Guard service, which uses custom heuristics and system prompt instructions to reject off-topic, malicious, or adversarial inputs (such as system prompt injection attempts) before invoking any downstream LLM or database queries. Next, a deterministic Project Detector parses the input using regular expressions and case-insensitive aliases to identify if the user is asking about a specific project namespace (like `classsync` or `talentforge`). If a namespace is identified, retrieval is filtered to that project; otherwise, it defaults to a global search. If query rewriting is enabled, the query is then sent to the Query Rewriter (powered by Gemini). The rewriter evaluates if the user's input is ambiguous or too terse and rephrases it to optimize lexical and vector retrieval recall (for instance, transforming "why redis" into "explain the technical decisions and tradeoffs of using Redis caching in the project").
