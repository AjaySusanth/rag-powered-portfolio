# RAG-Powered Developer Portfolio - Retrieval Evaluation and Diagnostic Report

## Overview of the Offline Evaluation Framework in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, a dedicated offline evaluation framework was developed to measure the quality, accuracy, and reliability of the retrieval pipeline. Testing retrieval modifications (such as changing chunk boundaries, tuning weights, or adjusting prompts) without an objective measurement baseline makes it impossible to prevent regressions. To resolve this, the framework uses a structured Golden Dataset (`scalable-n8n-deployment-architecture.json` containing 15 multi-disciplinary questions) with manually mapped ground-truth sources. The evaluation framework automatically executes queries against the retrieval engine and computes standard informational retrieval metrics. Specifically, it tracks Hit Rate@5 (percentage of queries where at least one expected source is in the Top-5), Recall@5 (proportion of expected sources successfully retrieved), Mean Reciprocal Rank (MRR, evaluating how close the first relevant chunk is to the top rank), Average Similarity (cosine similarity scores of hits), and Unique Sources@5 (the variety of unique source files returned in the Top-5). This metric-driven testing suite ensures that all pipeline modifications are validated before deployment.

## Comparative Analysis of Retrieval Configurations in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, offline evaluations were conducted across multiple retrieval configurations to establish the optimal balance between dense vector search and lexical search. The comparative results demonstrate the quantitative impact of each retrieval stage on accuracy and source diversity:

| Retrieval Configuration | Hit Rate@5 | Recall@5 | MRR | Avg Unique Sources@5 |
| :--- | :---: | :---: | :---: | :---: |
| **Vector-Only Search** | 73.3% | 60.0% | 0.569 | 4.20 |
| **BM25 Lexical Search** | 73.3% | 63.3% | 0.358 | 4.20 |
| **Hybrid (RRF Fusion)** | 66.7% | 60.0% | 0.456 | 4.40 |
| **Hybrid + Source Diversification** | 66.7% | 60.0% | **0.467** | **5.00** |

Analysis of these figures shows that while Vector-Only search yields a strong hit rate, it is highly prone to duplicate chunk starvation, filling the Top-5 context with sections from the same file. BM25 Lexical search achieves a high recall but has a low MRR of 0.358, indicating that relevant files are often ranked lower because keyword density does not equate to semantic relevance. Combining the two in a Hybrid pipeline using Reciprocal Rank Fusion (RRF) and post-process Source Diversification maintains high accuracy while promoting specific code artifacts and raising the unique source variety to a perfect score of 5.00.

## Impact of the Source Diversification Stage in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, the addition of the Source Diversification stage immediately resolved context starvation caused by duplicate chunks. Early tests revealed that dense vector similarity search frequently retrieved multiple sequential segments from a single lengthy document (such as five chunks from `n8n-aks-platform/lessons-learned.md`), which dominated the context window and starved the LLM of specific code configurations. The Source Diversification post-processor groups RRF-fused chunks by their source path and retains only the highest-ranked chunk per file (up to a default limit of 3 chunks per file). In evaluations on the AKS deployment dataset, this deduplication allowed code templates like `terraform/modules/aks/outputs.tf` to be successfully promoted into the Top-5. This optimization boosted the Average Unique Sources@5 from 4.40 to a perfect 5.00 and improved the Mean Reciprocal Rank (MRR) from 0.456 to 0.467 by ranking unique, relevant files higher.

## Diagnostic and Failure Analysis in RAG-Powered Developer Portfolio
In the RAG-Powered Developer Portfolio project, diagnostic failure logs from the evaluation runs are analyzed to identify specific query types that fail to retrieve their expected sources. In initial tests, queries seeking highly detailed infrastructure settings (e.g. asking about Helm security settings, SecretProviderClass manifests, or KEDA HPA scaling limits) frequently failed to retrieve the actual template YAML files. Instead, high-level manual design files like `architecture.md` or `decisions.md` dominated the Top-5 due to the high density of conceptual keywords. These failure analyses directly influenced several improvements: first, implementing a heading-aware two-pass chunker to prepend parent section titles to every chunk, ensuring code blocks retain architectural context; second, integrating an LLM query rewriter to split complex developer questions into search sub-queries; and third, implementing a retrieval grader to evaluate and prune off-topic chunks before sending the prompt to the generation layer.
