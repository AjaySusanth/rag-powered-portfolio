# Scalable n8n Architecture on AKS — Architectural Decisions (ADRs)

## ADR-001: Queue-Mode Architecture vs. Single-Process Monolithic Deployment in Scalable n8n Architecture on AKS

In the Scalable n8n Architecture on AKS project, the decision was made to deploy n8n in queue mode with three distinct process types (`n8n-main`, `n8n-webhook`, and `n8n-worker`) and a Redis message broker, rather than using the default single-container monolithic deployment. The context for this decision is that n8n's default deployment model runs all functions in a single container. Under heavy workloads, a single long-running or resource-intensive workflow execution can block the Node.js event loop. While the loop is blocked, the administration REST API becomes unresponsive, incoming webhooks fail or time out, and cron schedules are missed. Splitting the workloads isolates executing workflows onto dedicated worker pods, while webhook receivers handle HTTP traffic concurrently. The primary alternatives considered were a single-container deployment (which was rejected due to its lack of reliability under load) and a two-process split of main and workers (rejected because webhook ingestion remained coupled to the API, leaving it vulnerable to traffic spikes). The accepted tradeoffs of this decision include a significantly larger operational surface area (managing three Deployments, two StatefulSets, and a Redis instance), more complex distributed log tracing across components, a hard dependency on Redis availability, and an increased base compute cost with a minimum requirement of five running pods.

## ADR-002: Custom Helm Chart Authoring vs. Upstream Community Chart in Scalable n8n Architecture on AKS

For deployment management in the Scalable n8n Architecture on AKS project, the decision was made to write a custom, minimal Helm chart from scratch rather than using the community-maintained n8n Helm chart. The context is that while the community chart saves time, it is built to support a wide variety of deployment scenarios (multiple databases, ingress controllers, and storage providers). This generality introduces complex, bloated templates that make security auditing and hardening difficult to verify. By authoring a custom Helm chart, the platform maintains absolute control over all manifests, facilitating zero-trust hardening policies directly in the templates. The alternative considered was utilizing the community chart with extensive values overrides. This was rejected because upgrades to the upstream chart could silently override custom security contexts or introduce untested configurations. The consequences of this choice include accepting full ownership of chart correctness and upgrades, a larger upfront authoring time, and the lack of a community-maintained upgrade path. Upstream updates require manual testing and tag bumps, and custom manifests (such as KEDA `ScaledObject` and Key Vault `SecretProviderClass`) must be maintained manually.

## ADR-003: ArgoCD OCI Chart Source vs. Git Repository Path Source in Scalable n8n Architecture on AKS

For the GitOps workflow in the Scalable n8n Architecture on AKS project, the decision was made to package the Helm chart in CI, publish it to Azure Container Registry (ACR) as an OCI artifact, and configure ArgoCD to pull from the registry rather than referencing a Git repository path. The context is that Git-path sourcing syncs the cluster state on every Git commit in the target directory. If a developer pushes work-in-progress or syntactically invalid files, ArgoCD immediately syncs them, risking cluster downtime. Sourcing from an OCI registry adds an intermediate validation gate: charts are linted, packaged as versioned, immutable artifacts (`1.0.0-<git-sha>`), and published to ACR only after passing CI tests. The alternatives considered were standard Git-path sourcing (rejected due to the risk of deploying broken manifests) and maintaining a separate Helm chart repo (rejected as over-engineered for a single-platform architecture). The accepted tradeoffs include placing the CI pipeline and ACR directly in the deployment critical path (a CI failure blocks all updates), a slightly more complex OIDC credential exchange, and a GitOps loop that is one step removed from Git (ArgoCD tracks the OCI registry version rather than the Git commit directly).

## ADR-004: KEDA Event-Driven Queue Scaling vs. Metric-Based HPA in Scalable n8n Architecture on AKS

To manage pod scaling for workflow executions in the Scalable n8n Architecture on AKS project, the decision was made to use KEDA (Kubernetes Event-driven Autoscaling) with a Redis list length trigger (`bull:jobs:wait`) to scale the worker tier, rather than standard Kubernetes Horizontal Pod Autoscaling (HPA) based on CPU and memory metrics. The context is that CPU and memory utilization are lagging indicators for queue-mode n8n workers. A worker pod consumes minimal resources until it pulls a job from the queue; by the time resource usage spikes to trigger standard HPA scaling, a backlog has already accumulated, introducing workflow latency. KEDA monitors the queue depth directly, enabling proactive scaling. The alternatives considered were standard CPU-based HPA (rejected as too slow to respond to queue bursts) and HPA using custom metrics via Prometheus Adapter (rejected because KEDA offers native Redis triggers and requires less configuration overhead). The accepted tradeoffs include the operational overhead of installing and maintaining the KEDA operator in the cluster, managing a `TriggerAuthentication` credential object, and configuring `ignoreDifferences` in ArgoCD to prevent sync fights over replica counts.

## ADR-005: Azure Key Vault CSI Driver vs. Plain Kubernetes Secrets in Scalable n8n Architecture on AKS

For secret management in the Scalable n8n Architecture on AKS project, the decision was made to store all runtime credentials (`postgres-password`, `redis-password`, `n8n-encryption-key`) in Azure Key Vault and project them into pods via the Secrets Store CSI Driver, rather than committing encrypted or plain Kubernetes Secrets to the repository. The context is that plain Kubernetes Secrets are only base64-encoded and can be read by anyone with basic access to the repository or etcd database. The alternatives considered were Sealed Secrets (rejected because key rotation requires re-encrypting all secrets and managing cluster-side decryption keys) and External Secrets Operator (rejected because it syncs secrets into standard Kubernetes Secrets, leaving them exposed in etcd). The accepted tradeoffs include placing Azure Key Vault in the critical path for pod startup (if Key Vault is unreachable, pods fail to start), cloud provider lock-in to Azure's security model, the need to restart pods manually or configure rotation polling to pick up rotated secrets, and managing two distinct identities (GitHub Actions OIDC write identity and Kubelet read-only identity).
